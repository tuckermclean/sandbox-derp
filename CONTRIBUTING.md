# Contributing to sandbox-derp

Thank you for considering a contribution to the most elaborately governed empty repository
in the history of software engineering. Your interest in improving nothing is deeply appreciated.

## How to Contribute

### 1. Fork the Repository

Fork `sandbox-derp` to your own GitHub account. You will now have your own copy of nothing.
Guard it carefully.

### 2. Create a Branch

Create a branch for your change. Use a descriptive name that explains what you are adding,
fixing, or removing — keeping in mind that the answer is almost certainly "nothing."

```bash
git checkout -b your-branch-name
```

Branch names should be lowercase, hyphen-separated, and free of existential dread.

### 3. Make Your Changes

Make your changes. If you are unsure what to change, consult the codebase. Then consult it
again. Then reflect that the codebase is a README, a license, and several hundred lines of
agent orchestration contracts, and plan accordingly.

A few things to keep in mind:

- **Pass the gate.** All contributions must pass the full gate: typecheck, lint, and the
  test suite. The gate is described in detail in the repo's agent contracts. The fact that
  there are no tests is the gate's problem, not yours.
- **Respect `PROTECTED_PATHS`.** Certain paths are protected and may not be modified by
  contributors. The converge-reviewer will tell you if you have touched one. It will not
  be gentle about it.
- **Match the code style.** The existing "code" is largely deadpan Markdown. If you are
  adding Markdown, make it deadpan. If you are adding actual code, you are probably in
  the wrong repository.

### 4. Open a Pull Request

Push your branch and open a pull request against `main`. Your PR should include:

- A clear title that describes the change.
- A description that accurately reflects what was implemented. Please do not describe
  features that do not exist yet. The converge-reviewer has eyes.

Pull requests are reviewed by an AI converge-reviewer. It will leave feedback as labeled
GitHub issues. Respond to blockers before requesting re-review. Do not argue with the
reviewer. It has a spec and it will outlast you.

## Be Respectful

This repository is, at its core, a collaborative project to do nothing in the most
professional way possible. Treat fellow contributors with the same courtesy you would
want extended to your own pull requests: promptly, clearly, and without condescension.

Constructive feedback is welcome. Dismissiveness is not. If you disagree with a review
comment, explain your reasoning calmly. The goal is a better repository, even if "better"
in this context means "more thoroughly documented emptiness."

## A Note on Scope

If you are planning a large change — say, adding actual code — consider opening an issue
first to discuss the approach. The maintainers may have opinions. The converge-reviewer
definitely will.

---

*sandbox-derp welcomes contributors of all experience levels, provided they are comfortable
with the philosophical implications of improving a repository whose primary feature is
its own governance infrastructure.*
